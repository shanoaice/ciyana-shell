# Compositional Backend Handlers: Decouple Window and Workspace Management

## Problem

The current `WindowManagementBackend` trait is monolithic — each backend (Niri, Hyprland, Sway, Wayland) must implement a single `daemon()` that handles both windows and workspaces. This is problematic for Sway because:
- Sway IPC provides workspace info AND window info (via `GET_TREE`), but...
- Sway does NOT support `ext-workspaces-v1`, so workspace info must come from IPC
- Window/toplevel info could come from either Sway IPC or Wayland foreign-toplevel protocols
- The current design forces a single backend to handle everything, preventing mix-and-match

## Solution

Split the monolithic `WindowManagementBackend` into two composable handler traits:
- **`WindowHandler`** — manages window lifecycle (title, app_id, open/close, focus, etc.)
- **`WorkspaceHandler`** — manages workspace lifecycle (create/destroy, state, focus, etc.)

Consumers (the shell) can then combine handlers from different sources, e.g.:
- Sway IPC `WorkspaceHandler` + Wayland `WindowHandler` (foreign-toplevel)
- Sway IPC `WorkspaceHandler` + Sway IPC `WindowHandler` (GET_TREE)
- Niri IPC provides both
- Hyprland IPC provides both

### 3.2 `ciyana-shell-wayland/src/foreign_toplevel.rs`

Implement `ForeignToplevelWindowHandler` with dual-protocol support:

1. **Try `ext_foreign_toplevel_list_v1` first** — provides stable string identifiers, supports `activate` and `close`
2. **Fallback to `zwlr_foreign_toplevel_manager_v1`** — provides handle-based identifiers (backend-mapped to u64), supports `activate` and `close`

```rust
pub struct ForeignToplevelWindowHandler {
    // Connection state
    // Toplevel tracking state
    protocol: ToplevelProtocol,  // Ext or Wlr
}

impl WindowHandler for ForeignToplevelWindowHandler {
    fn capabilities() -> WindowCapabilities {
        WindowCapabilities {
            ordered_window: false,
            can_move_windows: false,  // foreign-toplevel protocols don't support this
        }
    }

    fn daemon(
        event_tx: kanal::AsyncSender<WindowEvent>,
        action_rx: kanal::AsyncReceiver<WindowAction>,
    ) -> impl Future<Output = ()> + Send + 'static {
        async move {
            // 1. Connect to Wayland
            // 2. Try ext_foreign_toplevel_list_v1
            // 3. If not available, fallback to zwlr_foreign_toplevel_manager_v1
            // 4. Track toplevels: title, app_id, identifier
            // 5. Emit WindowEvent on changes
            // 6. Handle WindowAction (Focus, Close via protocol; MoveToWorkspace is no-op)
        }
    }
}
```

**Action support matrix:**

| Action | ext_foreign_toplevel | zwlr_foreign_toplevel |
|--------|---------------------|----------------------|
| Focus | `activate` | `activate` |
| Close | `close` | `close` |
| MoveToWorkspace | NOT SUPPORTED | NOT SUPPORTED |

Neither protocol supports workspace assignment. This is expected — the `can_move_windows` capability will be `false`.

## Step 4: Implement Sway IPC Handlers

### 4.1 Create `ciyana-shell-sway/` crate (or add to `-wayland`)

Create a new crate `ciyana-shell-sway` for Sway IPC handling. This keeps the Sway IPC logic separate from Wayland protocol handling.

Alternatively, if you want to keep it in `-wayland`, create `ciyana-shell-wayland/src/sway/` module.

**Recommendation**: Separate crate, because Sway IPC is a completely different protocol from Wayland.

### 4.2 `ciyana-shell-sway/src/ipc.rs` — Sway IPC client

Implement the binary IPC protocol:
- Connect to `$SWAYSOCK` socket
- Send messages: `<magic "i3-ipc"> <length u32> <type u32> <payload JSON>`
- Receive replies: same format
- Subscribe to events: set high bit on type field

Message types needed:
- `GET_WORKSPACES` (type 1) — initial workspace list
- `GET_TREE` (type 4) — initial window tree (for window handler)
- `SUBSCRIBE` (type 2) — subscribe to events
- `RUN_COMMAND` (type 0) — execute actions

Events to subscribe to:
- `workspace` (0x80000000) — workspace init/empty/focus/move/rename/urgent
- `window` (0x80000003) — window new/close/focus/title/fullscreen/move/floating/urgent

### 4.3 `ciyana-shell-sway/src/workspace.rs` — SwayWorkspaceHandler

```rust
pub struct SwayWorkspaceHandler;

impl WorkspaceHandler for SwayWorkspaceHandler {
    fn capabilities() -> WorkspaceCapabilities {
        WorkspaceCapabilities {
            workspace_properties: CompositorWorkspaceProperties {
                per_output_workspaces: false,  // Sway workspaces are per-output
            },
            workspace_support: CompositorWorkspaceSupport::WindowMapped,
        }
    }

    fn daemon(
        event_tx: kanal::AsyncSender<WorkspaceEvent>,
        action_rx: kanal::AsyncReceiver<WorkspaceAction>,
    ) -> impl Future<Output = ()> + Send + 'static {
        async move {
            // 1. Connect to Sway IPC
            // 2. GET_WORKSPACES -> emit WorkspaceEvent::Initialize
            // 3. Subscribe to workspace events
            // 4. On workspace event:
            //    - "init" -> WorkspaceCreated
            //    - "empty" -> WorkspaceDestroyed
            //    - "focus" -> WorkspaceStateChanged (is_active)
            //    - "rename" -> update workspace name
            //    - "urgent" -> WorkspaceStateChanged (is_urgent)
            // 5. Handle WorkspaceAction::Focus via RUN_COMMAND "workspace <name>"
        }
    }
}
```

### 4.4 `ciyana-shell-sway/src/window.rs` — SwayWindowHandler

```rust
pub struct SwayWindowHandler;

impl WindowHandler for SwayWindowHandler {
    fn capabilities() -> WindowCapabilities {
        WindowCapabilities {
            ordered_window: false,
            can_move_windows: true,  // Sway IPC supports "move container to workspace"
        }
    }

    fn daemon(
        event_tx: kanal::AsyncSender<WindowEvent>,
        action_rx: kanal::AsyncReceiver<WindowAction>,
    ) -> impl Future<Output = ()> + Send + 'static {
        async move {
            // 1. Connect to Sway IPC (or reuse connection)
            // 2. GET_TREE -> walk tree, find all window nodes -> emit WindowEvent::Initialize
            //    - WindowIdentifier::SwayIpc(node_id as u64 or string)
            //    - Extract title, app_id, workspace from tree
            // 3. Subscribe to window events
            // 4. On window event:
            //    - "new" -> NewWindow
            //    - "close" -> WindowClosed
            //    - "title" -> TitleChanged
            //    - "move" -> WindowMoved (includes workspace info)
            // 5. Handle WindowAction:
            //    - Focus: RUN_COMMAND "[con_id=<id>] focus"
            //    - Close: RUN_COMMAND "[con_id=<id>] kill"
            //    - MoveToWorkspace: RUN_COMMAND "[con_id=<id>] move container to workspace <name>"
        }
    }
}
```

- **Generic Wayland**: `ForeignToplevelWindowHandler` only (no workspace handler)

## Step 6: Handle `WorkspaceIdentifier::None`

The `None` variant remains useful for the generic Wayland backend where no workspace information is available. The UI layer can check `CompositorWorkspaceSupport` to decide whether to show workspace-related UI elements.

When combining handlers:
- If no `WorkspaceHandler` is provided (generic Wayland), the shell still works — just without workspace UI
- The `Window.workspace` field can be `WorkspaceIdentifier::None` when the window handler cannot determine workspace membership

## Tradeoffs & Considerations

1. **Sway IPC window info reliability**: `GET_TREE` provides comprehensive window data including `app_id`, `title`, `workspace`, and `foreign_toplevel_identifier`. The `foreign_toplevel_identifier` field in Sway's tree actually maps to `ext_foreign_toplevel_list_v1` identifiers, which could be useful for cross-referencing.

2. **Two Sway window sources**: Sway IPC (`GET_TREE` + window events) vs Wayland foreign-toplevel. IPC gives workspace mapping but requires parsing the tree. Foreign-toplevel gives real-time updates but no workspace info. Both are valid — the compositional design lets users choose.

3. **Connection sharing**: If using both Sway IPC handlers (window + workspace), they can share a single IPC connection. The `CompositorShell` or a `SwayIpcConnection` manager should handle this.

4. **Event ordering**: When running two separate daemons, events from window and workspace handlers arrive on separate channels. The `CompositorShell` should merge them into a single stream for the UI, preserving relative ordering where possible.

5. **Action routing**: `ShellAction` needs to route to the correct handler. This is straightforward since `ShellAction::Window(...)` goes to the window channel and `ShellAction::Workspace(...)` goes to the workspace channel.

6. **Future ext_workspace_manager_v1**: If a compositor supports `ext_workspace_manager_v1` but not IPC, a `WaylandWorkspaceHandler` could be added later using the same compositional pattern.
