# Plan: Window Management Backend Architecture

## Decision: Combine Workspaces + Windows into One Module

**Recommendation:** Yes, combine them into a single `window_management` module with one `WindowManagementBackend` trait.

### Rationale

1. **Shared daemon task** - Both windows and workspaces are monitored from the same async event loop. Splitting them would require two daemons that need to coordinate state.

2. **Coupled events** - Workspace membership changes, window focus, and workspace activation are inherently related. Emitting them from one source avoids race conditions.

3. **Simpler capability detection** - One place to query "does this compositor support workspaces?" rather than checking two separate backends.

4. **Existing pattern** - `WindowBackend` already exists; extending it is cleaner than creating a parallel `WorkspaceBackend`.

## Implementation Plan

### 1. Restructure Core Types

**File:** `ciyana-shell-core/src/window_management.rs` (rename from `window.rs`)

#### Workspace Group Model

Compositors handle workspace-to-output relationships differently:

- **Niri**: One workspace group per output (monitor), each with its own set of workspaces
- **Hyprland**: One workspace group spanning all outputs, shared workspaces move between monitors

We model this with `WorkspaceGroup` to represent a set of outputs that share workspaces.

#### Types

```rust
/// Opaque output identifier, backend-specific
pub struct OutputId(pub u64);

/// Opaque group identifier, backend-specific
pub struct WorkspaceGroupId(pub u64);

/// Opaque workspace identifier, backend-specific
pub struct WorkspaceId(pub u64);

/// A workspace group bound to a set of outputs
pub struct WorkspaceGroup {
    pub id: WorkspaceGroupId,
    pub output_ids: Vec<OutputId>,  // which monitors this group covers
}

/// A single workspace within a group
pub struct Workspace {
    pub id: WorkspaceId,
    pub group_id: WorkspaceGroupId,  // which group this belongs to
    pub name: String,
    pub state: WorkspaceState,       // active, urgent, hidden
}

bitflags! pub struct WorkspaceState: u8 {
    const ACTIVE = 1;
    const URGENT = 2;
    const HIDDEN = 4;
}
```

#### Events (renamed from `WindowEvents` to `WindowManagementEvents`)

```rust
pub enum WindowManagementEvents {
    // Window events (existing)
    Initialize(Vec<Window>),
    TitleChanged(WindowIdentifier, String),
    AppIdChanged(WindowIdentifier, String),
    NewWindow(Window),
    WindowClosed(WindowIdentifier),

    // Workspace group events
    WorkspaceGroupCreated(WorkspaceGroup),
    WorkspaceGroupDestroyed(WorkspaceGroupId),

    // Workspace events
    WorkspaceCreated(Workspace),
    WorkspaceDestroyed(WorkspaceId),
    WorkspaceStateChanged(WorkspaceId, WorkspaceState),

    // Window-to-workspace membership
    WindowEnteredWorkspace(WindowIdentifier, WorkspaceId),
    WindowLeftWorkspace(WindowIdentifier, WorkspaceId),
}
```

#### Actions (renamed from `WindowAction` to `WindowManagementAction`)

```rust
pub enum WindowManagementAction {
    Focus(WindowIdentifier),
    Close(WindowIdentifier),
    FocusWorkspace(WorkspaceId),
    MoveWindowToWorkspace(WindowIdentifier, WorkspaceId),
}
```

### 2. Add Capability Detection

```rust
pub struct WindowManagementCapabilities {
    pub workspaces_supported: bool,
    pub workspace_creation: bool,
    pub window_workspace_assignment: bool,
    /// True if workspace groups are per-output (Niri-style),
    /// false if workspaces are shared across all outputs (Hyprland-style)
    pub per_output_workspaces: bool,
}

pub trait WindowManagementBackend {
    fn capabilities() -> WindowManagementCapabilities;
    fn daemon(
        event_sender: kanal::AsyncSender<WindowManagementEvents>,
        action_receiver: kanal::AsyncReceiver<WindowManagementAction>,
    ) -> impl Future + Send + 'static;
}
```

### 3. Compositor-Specific IPC Backends

**Strategy:** Use native IPC protocols for Niri and Hyprland instead of generic Wayland protocols. This gives full access to window-workspace mapping, which `ext_workspace_manager_v1` does not expose.

#### Niri Backend

**File:** `ciyana-shell-wayland/src/window_management/niri.rs`

- Uses Niri's Unix socket IPC (JSON over socket)
- Receives full window list with workspace assignment on connect
- Subscribes to events: `WindowOpened`, `WindowClosed`, `WorkspaceActivated`, `WindowMovedToWorkspace`
- `per_output_workspaces: true` - each output has its own workspace group
- `WorkspaceGroupId` maps to output ID, `WorkspaceId` maps to Niri's workspace ID

#### Hyprland Backend

**File:** `ciyana-shell-wayland/src/window_management/hyprland.rs`

- Uses Hyprland's Unix socket IPC (JSON over socket)
- Subscribes to event socket for: `openwindow`, `closewindow`, `workspace`, `movewindow`
- `per_output_workspaces: false` - single workspace group across all outputs
- `WorkspaceGroupId` is a constant (single group), `WorkspaceId` maps to Hyprland's workspace ID

#### Generic Wayland Fallback (Optional, Future)

**File:** `ciyana-shell-wayland/src/window_management/wayland.rs`

- Uses `ext_foreign_toplevel_list_v1` + `ext_workspace_manager_v1`
- Only works on compositors that support both protocols AND expose window-workspace mapping
- `workspaces_supported: false` if mapping is unavailable

### 4. Backend Discovery

At startup, the binary detects which compositor is running:

```rust
// Check environment variables
if std::env::var("NIRI_SOCKET").is_ok() {
    // Use Niri backend
} else if std::env::var("HYPRLAND_INSTANCE_SIGNATURE").is_ok() {
    // Use Hyprland backend
} else {
    // Try generic Wayland fallback
}
```

### 5. GUI Integration

- GUI queries `capabilities()` at startup to show/hide workspace-related UI
- If `workspaces_supported: false`, dock shows all windows (current behavior)
- If `workspaces_supported: true`, dock filters to active workspace windows only
- GUI maintains a `HashSet<WindowId>` of active-workspace windows by tracking enter/leave events
- **Status bar** uses `per_output_workspaces` to decide layout:
  - `true` (Niri): Each monitor's status bar segment shows its own workspace indicators
  - `false` (Hyprland): Single shared workspace indicator row for all monitors
- Status bar subscribes to `WorkspaceGroupCreated`/`Destroyed` to know which outputs have workspace groups, then listens to `WorkspaceCreated`/`StateChanged` within each group

### 6. Module Organization

```
ciyana-shell-core/src/
  window_management.rs    (types + trait)
  mod.rs                  (exports)

ciyana-shell-wayland/src/
  window_management/
    mod.rs                (backend dispatch, discovery)
    niri.rs               (Niri IPC implementation)
    hyprland.rs           (Hyprland IPC implementation)
    wayland.rs            (generic Wayland fallback, optional)
  mod.rs
```

## Tradeoffs Considered

| Approach | Pros | Cons |
|----------|------|------|
| **Native IPC** | Full access to window-workspace mapping, reliable | Compositor-specific code |
| **Generic Wayland protocols** | Portable | No window-workspace mapping in standard protocols |

**Verdict:** Native IPC wins because window-workspace mapping is essential for the dock feature and Wayland protocols don't provide it.
